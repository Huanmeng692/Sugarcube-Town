这是一个用VSCode编写Sugarcube游戏的，推荐的文件结构。
当然这并非必须，理论上除了res文件夹内的外部资源外，其他所有内容你都可以写在同一个twee文件内。
但为了能够清晰的整理自己的思路，便于后期编辑使用，也为了方便搬运大佬们的轮子，还是推荐按照这样的结构建立文件。

在此之前，你可能需要先做一些[准备工作](./准备工作.md)

如果你已经做完了准备工作，那就继续向下看吧。
接下来分别讲解各部分的作用分别是什么。

# [VSCode编辑器配置文件](../.vscode/settings.json)

用于按照你的习惯设置VSCode的行为和外观，就比如示例中所示。
{
	"editor.insertSpaces": false,
	"files.trimTrailingWhitespace": true,
	"files.trimFinalNewlines": true
}
作用分别是
（当你按下Tab键时，插入换行符而非空格）、
（保存文件时，删除每一行行尾多余的空格）、
（保存文件时，删除文件末尾多余的空行）
并不是必须的，但可以让你的VSCode更加顺手。
当然，如果你用的不是VSCode，或者觉得默认的设置就已经很好了，那就不用管它了。

# [twee配置文件](../ST.twee-config.json)

在Sugarcube中，除了使用
<<widget "宏名">><</widget>>
创建宏之外，还可以用JS的方式创建宏。
但如果不写配置文件，编辑器无法识别你用JS创建的宏。
这会导致你使用这个宏的时候，编辑器用黄色波浪线标注警告，虽然不会影响正常使用，但总归还是很烦的。
要想消除这种很烦的情况，可以在你的游戏根目录创建一个（*.twee-config.json）的文件，在里面加入你的宏名称。
虽然示例文件中添加了描述字符串，但那并不是必须的，最小的版本如下所示：
{
	"sugarcube-2": {
		"macros": {
			"example": {}
		}
	}
}
其中example为你的宏的名称，按照你自己的情况进行修改即可。
关于更多的信息，可以看[这里](https://github.com/cyrusfirheir/twee3-language-tools/?tab=readme-ov-file#custom-macro-definitions-for-sugarcube)

# [游戏资源](../res/)

Sugarcube游戏会把所有文本内容保存在HTML文件内，但图片，音频，视频之类的就需要你单独放在一起了。
你可以把不同类型的文件都放在一个文件夹内，也可以按照类型分别放置。
当然，如果你的游戏是纯文本的话，也可以不需要这些。

# [游戏源码](../src/) 【【【只有这个是必须的，重点看下面“故事片段”里提到的两个文件】】】

这是最重要的一部分，你的游戏所有显示的文本，所有功能性代码，还有所有的样式都应该保存在这里，以便构建HTML文件。
其中包括：

## [层叠样式表](../src/CSS/)

用于放置所有CSS样式，可以在里面创建细分的更多文件夹分别存放各种功能的样式表。

## [Javascript脚本](../src/Javascript/)

用于放置各种Javascript脚本，同样可以在里面创建更多细分的文件夹。

## [故事片段](../src/Passage/)

用于放置故事片段文件，后缀名为（.tw）或（.twee），同样可以创建更多细分的文件夹（如示例中分为了Config配置、Dialog弹窗、Story故事片段三个文件夹）。
推荐使用（.tw）文件，文件第一行以两个英文冒号加一个空格开头（:: ）后面跟你的片段名。
如果你的片段有标签，就在片段名后面加上空格，将标签写在英文方括号内。

需要注意的是，其中有两个文件是必须的。
一个是StoryTitle，一个是StoryData。
StoryTitle片段用于保存你的游戏的名字，也就是你这个游戏叫什么。
StoryData片段用于保存你游戏最基础的数据。
{
  "ifid": "11440488-86AA-429D-B676-D8623BC6CDF8",
  "format": "SugarCube",
  "format-version": "2.37.3",
  "start": "Start"
}
从上到下分别为：
你的游戏的“识别码”，将你的游戏的存档和其他游戏的存档区分开，越随机越好，你可以直接复制我的，然后随便把里面的一些字符改掉后使用。
你的游戏的故事格式，你创作的是Sugarcube游戏，这里就是Sugarcube。
你的故事格式的版本。
打开你的游戏，浏览器从哪个片段开始显示给玩家。

## [临时用一下的文件](../src/Temp/)

在创作游戏的过程中，你可能会经常需要调试一些功能和变量，这些临时使用的片段不会也不应该遗留到玩家的手中。
因此你可以把这些调试用的东西放在一个临时的文件夹里，等到发布游戏的时候将这个文件夹删除后重新Build成正式版的HTML文件，可以节省很多工作。
